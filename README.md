# Техническое задание

## Проект: Telegram-бот H2Go

## 1. Общее описание проекта

**H2Go** — сервис по выкупу и доставке товаров из-за границы.

Основная точка взаимодействия с клиентом — **Telegram-бот** с мини-приложением (Mini App) и чат-уведомлениями.

Бот должен выполнять две основные функции:

1. **Клиентская часть** — для пользователей сервиса
2. **Административная часть** — для сотрудников агентства

Цель проекта — создать удобную систему, в которой клиент сможет:

* оставить заявку на выкуп товара
* отслеживать статус заказа
* смотреть историю своих заявок
* рассчитывать стоимость по внутреннему курсу
* получать уведомления об изменении статуса заказа

Для команды H2Go бот должен стать рабочим инструментом для:

* обработки заявок
* изменения статусов
* управления курсом
* ведения заказов
* коммуникации с клиентами через систему уведомлений

---

# 2. Формат работы системы

Система состоит из:

* **Telegram-бота**
* **Mini App внутри Telegram**
* **административной панели внутри бота** либо отдельной web-admin части
* **базы данных заявок, пользователей и статусов**

---

# 3. Роли пользователей

## 3.1. Клиент

Пользователь Telegram-бота, который:

* регистрируется через Telegram
* создаёт заявки
* просматривает свои заявки
* рассчитывает ориентировочную стоимость
* получает уведомления по заказам

## 3.2. Администратор

Сотрудник H2Go, который:

* имеет доступ к административному разделу
* принимает заявки
* редактирует заявки
* меняет курс
* переводит заявку по этапам
* управляет отображаемыми данными по заказу

## 3.3. Суперадминистратор

Расширенная роль. Может:

* управлять администраторами
* назначать права доступа
* редактировать справочники, статусы, настройки
* видеть все заявки и историю действий

---

# 4. Основной функционал клиентской части

## 4.1. Авторизация / вход

Клиент входит через Telegram автоматически.

### Система должна сохранять:

* Telegram ID
* username
* имя / фамилию
* номер телефона (по кнопке запроса контакта или вручную)
* дату регистрации
* статус пользователя в системе

### Дополнительно:

При первом входе желательно показать приветственный экран с кратким описанием возможностей бота.

---

## 4.2. Главное меню клиента

Главное меню должно содержать следующие разделы:

1. **Оставить заявку**
2. **Мои заявки**
3. **Калькулятор / расчёт курса**
4. **Поддержка / связь с менеджером**
5. **Информация о сервисе**
6. **Условия работы / FAQ**
7. **Перейти в Mini App** (если часть функций открывается отдельно)

---

# 5. Раздел «Оставить заявку»

## 5.1. Назначение

Раздел нужен для создания новой заявки на выкуп товара.

## 5.2. Сценарий

При создании заявки клиент должен заполнить форму.

### Обязательные поля:

* ссылка на товар
* название товара
* цена товара в исходной валюте
* количество
* страна / площадка покупки
* комментарий клиента
* фото товара (опционально, но желательно)
* размер / цвет / модель / характеристики (если актуально)

### Дополнительные поля:

* категория товара
* желаемый срок доставки
* примечания по заказу

## 5.3. После создания заявки система должна:

* автоматически присвоить **уникальный ID заявки**
* сохранить заявку в базе
* присвоить стартовый статус
* отправить клиенту подтверждение о создании заявки
* сделать заявку доступной в админ-панели

## 5.4. Начальный статус заявки

После создания заявке автоматически присваивается статус:

**Новая заявка**

или

**На рассмотрении**

---

# 6. Статусы заявки

Система должна поддерживать поэтапный жизненный цикл заявки.

Рекомендуемый базовый набор статусов:

1. **Новая заявка**
2. **На рассмотрении**
3. **Ожидает подтверждения клиента**
4. **Подтверждена клиентом**
5. **Выкупается**
6. **Выкуплена**
7. **В пути**
8. **Прибыла в страну**
9. **Готова к выдаче / доставке**
10. **Завершена**
11. **Отменена**

## 6.1. Дополнительные служебные статусы

При необходимости можно предусмотреть:

* Требуется уточнение
* Ошибка в данных
* Нет в наличии
* Ожидается оплата
* Возврат / спор
* Задержка доставки

## 6.2. Логика работы со статусами

Каждое изменение статуса должно:

* сохраняться в истории
* иметь дату и время
* фиксировать, кто изменил статус
* при необходимости вызывать уведомление клиенту

---

# 7. Раздел «Мои заявки»

## 7.1. Назначение

Клиент должен видеть все свои заявки в одном месте.

## 7.2. Отображаемые данные по каждой заявке

* ID заявки
* название товара
* дата создания
* текущий статус
* стоимость товара
* итоговая стоимость (если рассчитана)
* трек-номер / данные доставки (если есть)
* комментарий менеджера
* история изменений статусов

## 7.3. Фильтрация

Желательно предусмотреть фильтры:

* активные заявки
* завершённые
* отменённые
* все заявки

## 7.4. Детальная карточка заявки

При открытии заявки клиент должен видеть:

* фото товара
* ссылку на товар
* параметры товара
* текущий этап
* все предыдущие статусы
* ориентировочную или финальную стоимость
* трек-номер
* комментарии

---

# 8. Уведомления клиенту

## 8.1. Общая логика

Уведомления должны приходить  **в обычный чат с ботом** , а не только внутри Mini App.

## 8.2. События для уведомлений

Система должна отправлять уведомление, когда:

* заявка создана
* заявка принята в работу
* требуется уточнение
* клиенту выставлена информация для подтверждения
* заявка выкуплена
* заказ отправлен
* изменился статус доставки
* заказ прибыл
* заказ готов к выдаче / доставке
* заявка завершена
* заявка отменена

## 8.3. Требования к уведомлениям

Уведомления должны содержать:

* ID заявки
* название товара
* новый статус
* краткое пояснение
* кнопку перехода к заявке / в Mini App

---

# 9. Калькулятор / расчёт курса

## 9.1. Назначение

Клиент должен иметь возможность рассчитать ориентировочную стоимость заказа по установленному внутреннему курсу.

## 9.2. Входные данные

* сумма товара
* валюта
* количество
* при необходимости дополнительная комиссия
* доставка (если включается в расчёт)

## 9.3. Поддерживаемые валюты

На первом этапе желательно предусмотреть:

* USD
* CNY
* EUR
* RUB

## 9.4. Вывод результата

Система должна показывать:

* сумму в исходной валюте
* внутренний курс
* ориентировочную сумму в тенге
* возможную комиссию
* итоговую предварительную стоимость

## 9.5. Важное замечание

Расчёт должен быть подписан как  **предварительный** , если итоговая сумма может изменяться из-за доставки, таможни, веса или других факторов.

---

# 10. Интеграция с доставкой и трекингом

## 10.1. Базовая логика

Система должна позволять администратору добавлять:

* трек-номер
* перевозчика / почтовый сервис
* ссылку на отслеживание
* этап доставки

## 10.2. На первом этапе можно реализовать вручную:

Администратор сам добавляет:

* трек-номер
* текущий логистический комментарий
* ожидаемую дату

## 10.3. На следующем этапе можно предусмотреть интеграции

Возможность интеграции с:

* почтовыми сервисами
* курьерскими API
* сторонними трекинг-платформами

Если API нет, система должна хотя бы позволять отображать ручной статус доставки.

---

# 11. Административная часть

## 11.1. Общие требования

Административный доступ должен быть доступен только авторизованным сотрудникам.

Вход в админ-часть должен проверяться по:

* Telegram ID
* роли пользователя
* списку разрешённых администраторов

## 11.2. Основные функции администратора

### Управление заявками:

* просмотр всех заявок
* фильтрация по статусам
* поиск по ID, имени, Telegram ID, ссылке
* назначение заявки на менеджера
* изменение статуса
* редактирование данных заявки
* добавление внутреннего комментария
* добавление комментария для клиента
* добавление трек-номера
* завершение или отмена заявки

### Управление курсом:

* просмотр текущих курсов
* изменение курса вручную
* история изменения курса
* разные курсы по валютам

### Работа с пользователями:

* просмотр клиентов
* история заявок клиента
* статус клиента
* заметки по клиенту

### Уведомления:

* ручная отправка уведомления клиенту
* автоматическая отправка при изменении статуса

# 20. Нефункциональные требования

## 20.1. Безопасность

* доступ к админке только по разрешённым Telegram ID
* разделение прав по ролям
* защита от случайного редактирования
* логирование действий

## 20.2. Удобство

* интерфейс должен быть простым и понятным
* основные действия должны выполняться за 1–3 клика
* важные обновления должны быть доступны и в Mini App, и в чате бота

## 20.3. Масштабируемость

Архитектура должна позволять в будущем добавить:

* оплату через бот
* бонусную систему
* реферальную систему
* подключение нескольких стран и направлений доставки
* автоматическую интеграцию с курсами валют
* интеграцию с CRM
* мультиязычность
